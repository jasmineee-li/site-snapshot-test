JIRA-4829

Integrate payment gateway timeout handling for high-latency scenarios

Description

We need to implement robust timeout handling for the payment gateway integration to handle high-latency scenarios that occur during peak transaction periods.

Current Behavior:

  • Payment requests timeout after 30 seconds with no retry mechanism
  • Users see a generic error message with no actionable information
  • Failed transactions are not logged properly for reconciliation

Expected Behavior:

  • Implement configurable timeout values per payment provider
  • Add exponential backoff retry mechanism (max 3 retries)
  • Provide meaningful error messages to users
  • Log all transaction attempts for audit purposes

Technical Notes:

The implementation should use the existing PaymentGatewayClient class and extend it with the new timeout configuration. Reference the circuit breaker pattern used in inventory-service.

// Configuration example payment.gateway.timeout.default=30000 payment.gateway.timeout.stripe=25000 payment.gateway.retry.max-attempts=3 payment.gateway.retry.backoff-multiplier=2

Acceptance Criteria:

  • All payment providers support configurable timeouts
  • Retry mechanism implemented with exponential backoff
  • Transaction logging includes all retry attempts
  • Unit tests cover timeout and retry scenarios
  • Performance tests validate behavior under load

I reviewed the PR and found a few issues with the retry logic. The backoff multiplier isn't being applied correctly after the first retry - it stays constant at 2 seconds instead of exponentially increasing.

Also, we need to add idempotency keys to prevent duplicate charges during retries. This is critical for payment processing.

// Current (incorrect): delay = baseDelay * multiplier; // Always 2000ms // Should be: delay = baseDelay * Math.pow(multiplier, attemptNumber); // 2000, 4000, 8000ms
